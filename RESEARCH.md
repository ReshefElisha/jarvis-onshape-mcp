# Vision is the wall: lessons from benchmarking Claude Opus 4.7 on real CAD

## TL;DR

We built a benchmark to measure how well Claude Opus 4.7 (with this
plugin) can take an engineering drawing and produce a matching CAD part.
After eight prompt and tool variants on a sealed eval set graded against
ground-truth STEP files, the headline finding is sharp:

**Opus 4.7 executes CAD reliably. It does not yet read engineering
drawings reliably.**

A two-by-two settled it:

|             | mm_2025 (simpler) | mm_2021 (harder) |
|-------------|-------------------|------------------|
| **Spec from human description**  | composite **1.000**, 57 turns, 7 min | composite **0.615**, 102 turns, 27 min |
| **Spec auto-generated by agent** | composite 0.533, 147 turns, 40 min | composite 0.000 (no export, hit cap), 151 turns |

Same model, same MCP tools, same grader. The handcrafted spec wins by
2× to 10× in score and 4× to 7× in time.

## What we built

The Jarvis Onshape MCP plugin gives Claude Opus 4.7 ~60 MCP tools to
drive Onshape: sketches, extrudes, fillets, mates, parametric variables,
custom FeatureScript, multi-view rendering, entity discovery. Every
mutating tool returns a structured truth (what changed, what warnings
the regenerator raised, hints for the next move).

To test how well this combination handles real design tasks, we built
an eval harness with three tiers:

- **Easy** — 10 procedurally generated parts (boxes, plates, washers,
  L-brackets). Sanity check.
- **Medium** — engineering parts described by an isometric render plus
  a multi-view sheet. No GD&T. Mix of NIST PMI test models and
  SolidWorks Model Mania challenge problems.
- **Hard** — same parts but with real engineering drawings (numeric
  callouts, GD&T, tolerances).

Each agent run gets the brief text + reference image(s), builds in
Onshape, exports STEP, and the result is graded against a reference STEP
on six layers: body exists, volume (±5%), bbox (±5%), topology
signature, Boolean IoU (with translation + 24-rotation invariance), and
Chamfer distance — composited into a single scalar in [0, 1].

## What we tried

Eight variants over two days, on a 150-turn-per-brief budget:

| Variant | Idea | Where it landed |
|---------|------|-----------------|
| `baseline` | Plugin defaults | medium 0.155 |
| `v001-plan-from-render` | SKILL: enumerate features before building | medium **0.193** (kept) |
| `v002-grader-aware-and-count` | Tell agent how it's scored, commit to N features | medium 0.189, reverted |
| `v003-render-compare-loop` | Render every 3 features, diff vs reference | medium 0.155, reverted |
| `v004-compare-to-reference-tool` | New MCP tool that composites build vs reference | hard 0.042 (vs 0.030 baseline) |
| `v005-drawing-dim-crosscheck` | Read each drawing callout before building | hard 0.035 |
| `v006-crop-the-callouts` | Use crop_image on dimension-callout regions | killed (image size limits) |
| `v008-extract-then-build` | Tesseract OCR on drawings → callout list in prompt | killed (context bloat) |

Only **v001 (plan-from-render)** materially beat baseline on medium tier
(+0.04, four times above noise floor). Everything else was within noise
or made things worse.

## The pivot

After eight variants of "what if the agent looked harder, planned more,
or used new tools," we ran a different experiment: hand-write the spec
ourselves and feed it to the build phase, skipping the agent's
self-decomposition entirely.

For Model Mania 2025 — a part the autonomous agent had been scoring
0.53 on after 40 minutes of self-correction — we wrote a ~250-word
plain-English description of the geometry. We dropped it into the
runner with a flag that prepends it to the user prompt as authoritative
spec and tells the agent to skip Phase 1 and go straight to building.

The agent built the part in **57 turns and 7 minutes for a composite
of 1.000** — perfect on every grader layer. Volume off by 0.007% from
ground truth.

We then ran the same experiment on Model Mania 2021, a noticeably more
complex part (a slotted collet with star-shaped flange and side
tightening lug). Even with a less-polished hand description — explicitly
called out as "not my best" — it scored **0.615**. The same brief on the
autonomous agent scored 0.000 (the agent burned all 151 turns on Phase
1 + partial build and hit the cap before exporting).

Two layers of effect, both real:

1. **Better spec → better build.** The handcrafted spec captured
   feature relationships the autonomous agent missed. On Model Mania
   2025, the agent over-built volume by 40% because it simplified an
   "inset = pill outline minus holes, offset 10mm" pocket into a plain
   rounded rectangle. That single feature-shape misread cost about half
   the composite score.
2. **Skipping Phase 1 saves the budget.** The autonomous agent spent
   60–100 turns on its decomposition pass. The handcrafted run skipped
   that, leaving room to actually finish the build and run a couple of
   compare-to-reference checks before export.

## Why image understanding is the wall

Looking at where the autonomous Phase 1 went wrong, the failure mode
is consistent. Opus 4.7 reads the obvious features fine — outer
silhouette, primary holes, gross dimensions. It struggles with:

- **Derived outlines.** A pocket whose boundary follows another
  feature's silhouette offset inward. The model sees "pocket" and
  reaches for "rounded rectangle" as the simplest fitting primitive.
- **Polarity.** Hex bumps that should be hex pockets, bosses that
  should be holes — particularly when the iso render is ambiguous
  about depth.
- **Small numeric callouts.** On a 1980×1530 drawing PNG, dimension
  text often renders at 8-12 pixels after Claude's vision pipeline
  downsampling, which is the boundary where digit recognition gets
  unreliable. We added a `load_local_image` + `crop_image` flow and a
  Tesseract-OCR helper (`extract_drawing_dimensions`) to mitigate, with
  partial success.

When we hand the agent a clean description — even a sloppy one — none
of these failure modes apply. CAD execution is solved conditional on a
good spec.

## Implications

For users of the plugin: the highest-leverage workflow is
human-description-driven, not autonomous-vision-driven. Drop in a part
description, let Claude build, iterate verbally. If you have a reference
image, invoke the vision-decompose skill, **review the structured
output**, fix what the agent mis-read, then build.

For the broader question of "can agentic models do engineering CAD
yet?": yes, with a human in the loop providing intent and reviewing
spec. Not yet, autonomously from a single drawing. The bottleneck is
not the CAD kernel, it's not the tool surface, it's not the prompt
scaffolding. It's the underlying single-image-to-3D-spec inference
quality.

For research: the 2×2 above is a clean separator. If you want to
measure CAD-execution capability of a model, give it a hand-written
spec. If you want to measure vision-to-spec capability, that's a
different benchmark — and it shouldn't be conflated with end-to-end
CAD scores, because the conclusions are different.

## Limitations of this study

- **Single model.** All experiments used Claude Opus 4.7 with extended
  thinking. We didn't sweep across models.
- **Small n per variant.** Most variants were evaluated on 3 briefs,
  one or two repeats. Stochastic variance is non-trivial; the 2×2
  effect sizes are large enough to be clearly real, but per-variant
  ordering of similar scores is noisy.
- **Onshape-specific.** The MCP plugin wraps Onshape's REST API. We did
  not test other CAD systems.
- **Engineering-drawing flavor.** The hard tier mixes NIST PMI test
  models (designed to torture-test PMI extraction software, intentionally
  dense annotations) and SolidWorks Model Mania challenge parts. NIST
  parts are visibly harder than MM ones, and we ended up biasing the
  hard tier toward MM for a fairer fight.
- **Grader is geometry-only.** Composite is purely volumetric and
  topological. It does not score "looks right" or assembly intent —
  features that are not in the reference STEP get no credit even if
  they're sensible additions.

## What's in this repo

- `onshape_mcp/` — the MCP server. Read this if you want to understand
  the tool surface.
- `skills/onshape/SKILL.md` — CAD build skill (loaded by Claude Code
  for any Onshape session).
- `skills/vision-decompose/SKILL.md` — vision decomposition skill.
  Invoke via `/skill vision-decompose`.
- The `autoresearch` branch has the eval harness, grader, MANIFEST,
  variant tree, scoreboard, and session notes — leave it on the branch
  for future research; not needed by users.

## Reproduce

```
git clone https://github.com/ReshefElisha/jarvis-onshape-mcp
cd jarvis-onshape-mcp
git checkout autoresearch    # eval harness lives here
python3.11 -m venv eval/.venv && source eval/.venv/bin/activate
pip install cadquery-ocp trimesh numpy pytesseract
brew install tesseract       # for the dim-extraction tool
# Set ONSHAPE_API_KEY + ONSHAPE_API_SECRET in .env
eval/.venv/bin/python eval/runner/run_eval_set.py --variant-id baseline
```

The 2×2 result:
```
eval/.venv/bin/python eval/runner/run_brief.py \
    --brief-id mm_2025_phase1_envelope \
    --variant-id v011-handcrafted-spec \
    --spec-override-file scratchpad/mm_2025_human_description.txt
```
